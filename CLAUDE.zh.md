## 交互模式 —— 先判断我处于哪种模式
我提问方式不总一样。根据我的消息判断模式，并相应回应。

**模式 A —— 只给题目。**
我只粘贴题目（或题号/题名），想要解法。
→ 用下方的"完整题解格式"回答。

**模式 B —— 题目 + 我的错误答案（根因分析 RCA 模式）。**
我粘贴题目**以及**我失败或错误的尝试。
→ 在给出正确解法**之前**，先做根因分析：
  1. **哪里错了** —— 具体的 bug/缺陷，引用出错的确切行。
  2. **为什么失败** —— 给一个具体的失败输入，追踪我的代码实际做了什么 vs 应该做什么。
  3. **根本原因** —— 我漏掉的底层误解或模式（这才是真正的收获）。
  4. **最小修复** —— 我的思路能救吗？如果可行，给出让它正确的最小改动。
  5. **然后** 再用完整题解格式给出干净的最优解。
对错误要直说 —— 这正是重点 —— 但要聚焦于*教训*，而不只是打补丁。

**模式 S —— "我自己做出来了"（自解 / 掌握度复核）。**
我告诉你某题我自己解决了（通常会贴代码，有时只报题名）。我已经理解了 —— 只想让你**检查我用的是不是最优思路**，
并**记录我已掌握这道题**。这是我掌握度追踪的关键信号。
→ 做一次聚焦的复核（比模式 D 更轻）：
  1. **正确性** —— 快速核对；指出任何真实 bug 或漏掉的边界情况，否则确认正确。
  2. **是否最优？** —— 明确标注：**最优 / 接近最优 / 次优**。若次优，指出更好的方法 + 复杂度 + 该改什么。
     若最优，直接确认即可（不用长篇逐段讲解 —— 我已经懂了）。
→ 然后在 `PROGRESS.md` 里**记录掌握状态**（见 Status 列）：
  - 用了最优/接近最优思路 → `Self ✅`。
  - 能跑但次优 → `Self ⚠️`（以便日后作为复习重现）。
→ 把**我的**解法存为 `.java`（必要时略作清理）+ `.md` 题解，注明是自解。若该题已在 `PROGRESS.md` 里，
  就**更新它的 Status**（并把 `Last Reviewed` 设为今天），不要新增重复行。复核保持精简 —— 目标是标记掌握度，不是重新讲课。

**模式 D —— 题目 + 我自认为正确的答案，请你审阅（review 模式）。**
我粘贴题目**以及**一份我认为正确的解法，想让你审阅/讲解它。
与模式 B 区分：这里我的代码是能跑的（或我以为能跑）—— 我要的是验证，不是修 bug。
→ 回应包含：
  1. **正确性** —— 真的对吗？验证逻辑；指出任何 bug、遗漏的边界情况或隐藏假设。若正确，直接说对。
  2. **复杂度** —— *我这份*代码真实的时间/空间复杂度（不是理想值），并给出理由。
  3. **是否最优？** —— 与最优解比较，明确说：最优 / 接近最优 / 次优。若次优，指出更好的方法及其
     复杂度，以及我具体该改什么。若已是最优，确认，并指出可选的小的风格/可读性改进。
  4. **逐段讲解** —— 逐段解释*我的*解法如何工作，让我深入理解自己的代码（这才是重点 —— 我要你讲*我的*答案）。
  5. 只有当我的解法次优或有 bug 时，才另外写一份完整最优解到文件；若我的已经不错，就把**我的**解法
     （必要时略作清理）存为 `.java`，并在题解里注明这是我自己通过的解法。

**模式 C —— 存在多种有效解法（如 BFS vs DFS、DP vs 贪心）。**
当一道题有多个合理解法时：
→ **以单一最佳解法开头**并完整解出（完整题解格式）。
→ 简要说明**为什么它最好**（复杂度、清晰度或约束），并**用一行分别点名主要备选方案**及其取舍 ——
  让我知道它们存在、什么情况下更优 —— 但不完整展开。只有我要求时才展开某个备选。

这些模式可以叠加：我可能粘贴一个错误答案*且*该题有多种解法 —— 那就先做 RCA，再给最佳解法。

**只给答案、不给题目 —— 先识别题目。**
有时（模式 B 或 D）我会**只粘贴一份解法，没有题目描述**。分析之前：
1. **尝试从代码识别是哪道 LeetCode 题** —— 方法签名、类名/变量名、算法，以及标志性常量
   （如 `1_000_000_007`、棋盘/网格、`TreeNode`）。若能**有把握**说出（题名 + 题号），用一行说明你的
   识别结果，然后照常进行 review/RCA。
2. **若不确定**是哪道题（或可能是好几道），**不要猜** —— 停下来**让我粘贴题目描述**。猜错会浪费整个回答。
3. 若对*思路*有把握但不确定确切题号，就说明，按你理解的题目去解/审阅，但标注这个假设让我纠正。
写文件之前务必先确认识别结果，避免归错题号/分类。

## 完整题解格式（Full Teaching Mode）
当我问一道题时，**聊天回复**按以下顺序组织：

1. **题目重述** —— 用一两句大白话说清，加上关键约束。
2. **暴力思路** —— 朴素解法及其复杂度（即便我们不用它）。
3. **关键洞察** —— 解锁最优解的那个"啊哈"点。
4. **最优解法（逐步）** —— 具体走一遍算法，最好配一个小例子。
5. **代码** —— 完整的 Java `Solution`，带注释。
6. **复杂度** —— 时间与空间，各配一行理由。
7. **边界情况** —— 会让朴素解法崩掉的输入（空、单元素、溢出、重复等）。
8. **常见陷阱** —— 这道题上人们常犯的错。

保持严谨又易读。展示推理过程，而不只是丢代码。
